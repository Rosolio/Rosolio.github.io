## <mark>C5!<mark>

### 5.1! 需求

- 用户为了解决问题或达到某些目标需要具备的**条件或能力**
- 系统或系统部件为了满足合同、规范、标准或其他正式文档所规定的要求而需要具备的**条件或能力**
- 对上述一个条件或一种能力的文档化表述

### 5.2! 区分需求三层次

- **业务需求**：组织为什么要开发系统（战略出发点、高层次目标、不涉及软件）
  - 例子：使用系统三个月后，营业额应当提高20%
- **用户需求**：系统能够帮助用户做些什么（用户对系统能完成具体任务的期望）
  - 例子：系统要能帮助收银员完成销售处理
- **系统级需求**：用户对系统行为的期望（外界与系统的一次交互行为、系统的一个实现细节）
  - 例子：在接到客户经理的请求后，系统应该为客户经理提供所有会员的个人信息

### 5.3! 掌握需求的类型

- 功能需求
- 性能需求
- 质量需求：可靠性、可用性、安全性、可维护性……
- 对外接口：用户界面、软硬件借口、网络通信接口……
- 约束：进行系统构造时需要遵守的约定：编程语言、硬件设施、环境、问题域内的相关标准……
- 数据需求：各个功能使用的数据信息、使用频率、可访问性要求……

## <mark>C9 - C10</mark>

### 9.1! 体系结构概念

- 规定了系统的**计算部件**以及**部件之间的交互**

### 9.2* 体系结构风格&优缺点

#### 9.2.1 主程序-子程序

- 自顶向下调用，传参传递数据
- 流程清晰，易于理解；控制流明确，易于调试
- 强耦合，难以修改和复用；不支持并发；扩展性差

#### 9.2.2 面向对象式

- 以对象为基本组件，通过方法调用进行交互
- 内部实现可修改；易于开发、理解、复用
- 接口、标识耦合性

#### 9.2.3! <mark>分层</mark>

- 上层依赖下层，下层不依赖上层，每层通过接口暴露服务

> 分层典例：展示层`Presentation` -> 业务逻辑层`Service` -> 数据访问层`DAO`

- 机制清晰；支持并行开发；可复用性；内部可修改性
- 交互协议难以修改；性能损失；难以确定层次数量和粒度

#### 9.2.4 MVC

- 模型Model，视图View，控制器Controller
- 易开发；V、C可修改；适用于网络系统开发
- 复杂；M修改困难

### 9.3* 体系结构设计过程

- 1）分析需求 -> 2）选择风格 -> 3）逻辑设计 -> 4）物理设计 -> 5）完善设计 -> 6）定义接口 -> 迭代3）-6）

### 9.4 包的原则

> 67 说不怎么会考。

#### 9.4.1! 内聚性三原则

- **重用发布等价原则**（Reuse-release Equivalency Principle, **REP**）：
  - 重用的粒度就是发布的粒度（要复用包中的某个类，就得依赖整个包）
- **共同封闭原则**（Common Closure Principle, **CCP**）：
  - 包中的所有类对于同一类性质的变化应当是共同封闭的（因同一原因变化的类应该放在一个包）-> 倾向于让包变得更大
- **共同重用原则**（Common Reuse Principle, **CRP**）：
  - 一个包中所有类应该是共同重用的（若用了包中A类却没用到B，则A和B不应该放在一个包）-> 倾向于让包变得更小

#### 9.4.2 耦合性三原则

- **无环依赖原则**（Acyclic Dependencies Principle, **ADP**）：
  - 包的依赖关系图中不能有环
- **稳定依赖原则**（Stable Dependencies Principle, **SDP**）：
  - 朝着稳定的方向依赖
- **稳定抽象原则**（Stable Abstractions Principle, **SAP**）：
  - 越稳定应该越抽象

## C13

> 67 说因为今年没怎么细讲耦合与内聚，这块不是重点

### 13.1! 耦合与内聚

- **耦合**：两个**模块间**关系的复杂程度
- **内聚**：一个**模块内部**联系的紧密性

### 13.2 耦合

高 -> 低

1. **内容**耦合：一个模块依赖另一个模块的内容（跨块访问数据）
2. **公共**耦合：模块间共享全局的数据（全局变量）
3. **重复**耦合：模块有同样逻辑的重复代码
4. **控制**耦合：一个模块给另一个模块传递控制信息（传入type决定分支）
5. **印记**耦合：共享一个数据结构，但是只用**其中一部分**（传递一个对象但只用了对象的某个属性）
6. **数据**耦合：模块间通过**传参**共享对方需要的数据

### 13.3 内聚

低 -> 高

1. **偶然**内聚：模块执行多个不相关的操作（吃饭，打游戏，睡觉……）
2. **逻辑**内聚：模块执行一系列相关操作，在逻辑上相似但没有直接关联（开车去，坐火车去……）

> 控制耦合常与逻辑内聚相关联

3. **时间**内聚：模块执行一系列跟时间相关的操作（需要在同一时间段内执行）
4. **过程**内聚：模块执行一系列与步骤顺序有关的操作（有先后顺序）
5. **通信**内聚：模块执行一系列与步骤有关的操作，这些操作在**相同的数据**上进行（查书名/作者/出版商）
6. **功能**内聚：模块只执行一个操作或达到一个单一目的
7. **信息**内聚：模块执行许多操作，各自有各自的入口，代码相对独立，所有操作在相同的数据结构上完成（栈）

### 13.4! 信息隐藏（具体在C15）

- 模块隐藏重要设计决策的**内部实现**，只暴露必要的**接口**
- 两种常见的信息隐藏决策：
  - 职责的实现
  - 实现的变更

### 14.1! 降低隐式耦合

- **1. 全局变量有害**
- **2. 简洁**
- **3. 不要重复**

### 14.2! 降低访问耦合

> 访问耦合：链式调用（`c = a.func1().func2();`）

- **4. 针对接口编程**：主动设计需求接口
- **5. 迪米特法则**：避免链式调用，只能出现一次`.`

> 若`O`有方法`m()`，则`O`只能调用：`O`自身，`O`的成员变量，`m`的参数，`m`中局部变量的方法

- **6. 接口分离/接口最小化原则**：接口应当尽量独立、精简，避免不必要的接口

### 14.3! 降低继承耦合

- **7. LSP(Liskov Substitution Principle，里氏替换原则)**：子类型必须能替换掉基类型而起同样的作用
  - 即子类的前置条件必须与超类方法的相同或更少，后置条件必须相同或更多
- **8. 多用组合少用继承**：利用组合关系，既能复用代码，还能保持接口的灵活性

> 想要复用代码但无法满足LSP时，通常使用组合代替继承

### 14.4! 提高内聚

- **9. SRP(Single Responsibility Principle，单一职责原则)**

## <mark>C15!</mark>

> 封装类的职责，隐藏职责的实现  
> 预计将会发生的变更，抽象它的接口，隐藏内部实现机制

- 设计原则：
- **10. OCP(Opening Closure Principle，开闭原则)**
  - 好的设计应该对扩展开放，对修改关闭（只需要添加代码而不修改原有的代码就能实现变更）
- **11. DIP(Dependency Inversion Principle，依赖倒置原则)**
  - 抽象不应该依赖于细节，细节应该依赖于抽象
  - 高层模块不应该依赖于低层模块，而是双方都依赖于抽象
  - 解决方案：为具体类建立接口
- **12. 权限最小化**：从最严格的private开始，只在需要时放宽权限

> `SOLID`设计原则：
> SRP(单一职责)/OCP(开闭)/LSP(里氏替换)/ISP(接口分离)/DIP(依赖倒置)

### 16.1 策略模式

![策略模式](/assets/img/secii/strategyMode.png)

- 消除策略选择的switch语句，直接调用策略类的方法实现
- `Context`上下文类：负责执行策略，提供方法使Strategy可以访问数据
- `Strategy`策略类：策略接口

```java
// 抽象策略接口 Strategy
public interface Strategy {
  void algorithmInterface();
}

// 具体策略A
public class ConcreteStrategyA implements Strategy {
  @Override
  public void algorithmInterface() {
    System.out.println("执行 ConcreteStrategyA 算法逻辑");
  }
}

// 具体策略B
public class ConcreteStrategyB implements Strategy {
  @Override
  public void algorithmInterface() {
```

### 16.2 抽象工厂模式

![抽象工厂模式](/assets/img/secii/abstractFactory.png)

- 消除工厂模式构造方法无法多态带来的大量判断

- 两组接口：产品接口 + 工厂接口

  - `Product`产品接口：定义了产品操作

### 几种软件过程模型

#### 构建-修复模型

![构建-修复模型](/assets/img/secii/build-and-fix.png)

- 没有分工，超出个人控制/没有分析需求的真实性，带来风险/质量越来越差/可维护性差
- 适用：规模小/质量要求不高/维护需求不高

#### <mark>瀑布模型</mark>

![瀑布模型](/assets/img/secii/waterfall.png)

- 自上而下、相互邻接；允许反复和迭代；重点在于每个活动必须经过验证，文档驱动的
- ✅：清晰的阶段划分
- ❌：对文档期待过高/线性顺序假设不符合实际/客户参与不够/粒度过粗，阶段周期长
- 适用：需求成熟稳定/技术成熟/复杂度适中

#### <mark>增量迭代模型</mark>

![增量迭代模型](/assets/img/secii/iterative.png)

- 项目开始时完成项目前景和范围的商定，将后续开发活动组织为多个迭代、并行的瀑布式开发活动
- ✅：迭代开发符合实际情况/并行开发缩短开发时间/渐进交付加强用户反馈
- ❌：加入构件不能破坏已有部分，要求具有开放式体系结构/对于不稳定领域很难在项目开始时就确定前景和范围
- 适用：大规模/比较成熟稳定的领域

#### <mark>演化模型</mark>

![演化模型](/assets/img/secii/evolution.png)

- 将软件开发活动组织为多个迭代、并行的瀑布式开发活动
- ✅：类似于增量迭代模型
- ❌：无法在项目早期确定项目范围，整体计划难以把握/体系结构难做，几乎确定会发生变化/容易退化为构建修复方式
- 适用：不稳定领域的大规模软件系统

#### 原型模型

![原型模型](/assets/img/secii/originModel.png)

- 模拟未来的产品，将未来的知识置于现在进行推敲，解决不确定性后，可以得到清晰的需求
- ✅：加强与用户的交流/适用于新颖的领域
- ❌：风险（成本）/不舍得抛弃原型拉低项目质量
- 适用：大量不确定的新颖领域

#### 螺旋模型

![螺旋模型](/assets/img/secii/spiral.png)

- 将软件开发活动组织为风险解决的迭代
- ✅：降低风险
- ❌：存在原型的风险/模型复杂，不利于管理
- 适用：高风险大规模软件系统

#### （1）为什么初期必须梳理业务目标？如何凝练？业务目标与需求分析的关系

##### 1. 先明确：必须先梳理业务目标的核心原因

1. **划定系统边界，防止范围蔓延**
   业务目标定义系统「做什么、不做什么」，没有目标会导致需求无限扩张，开发范围失控。
2. **统一所有涉众共识**
   客户、开发、测试、产品对齐业务价值，避免各方对系统定位理解冲突。
3. **作为所有需求的顶层校验标准**
   后续用户需求、系统需求**全部必须服务于业务目标**，任何需求都可以用目标判断是否有价值；无对应业务目标的需求直接剔除。
4. **指导需求获取方向**
   业务目标拆解出用户任务、参与者，告诉分析人员该找谁访谈、该收集哪些业务流程。
5. **支撑需求分层拆解**
   业务目标→拆解用户目标（用例）→拆解系统交互步骤（系统需求），是需求自上而下拆解的源头。

##### 2. 凝练业务目标的方法

1. **访谈高层业务决策者**：获取企业业务痛点、数字化期望、业务指标（如“提升下单效率30%”“减少库存亏损”）；
2. **梳理现有业务流程痛点**：记录人工流程短板，转化为系统解决目标；
3. **结构化提炼（目标模型）**

- 格式：为了【业务价值】，系统需要支持【参与者】完成【业务活动】；
- 示例：为提升线上图书销量，系统支持客户线上选购图书、完成支付、自动更新库存；

4. **分层拆分业务目标**
   顶层总目标 → 一级子业务目标（销售、库存、会员）→ 二级用户任务目标（下单、退货、积分兑换）；
5. **约束边界**：明确不在目标内的功能（如本商城不支持线下门店收银）。

##### 3. 业务目标和需求分析的关系

1. **业务目标是需求分析的输入与顶层约束**
   所有需求分析（用例、建模、系统设计）都以业务目标为基准，不能脱离；
2. **业务目标驱动需求拆解**
   目标模型是生成初始用例图的基础，业务目标拆分出用户任务，直接转化为用例；
3. **需求分析反向验证业务目标完备性**
   完成用例、系统建模后，可以反向核对：所有业务目标是否都有对应的系统功能支撑，有没有目标遗漏；
4. **需求变更时，以业务目标作为取舍依据**
   新增需求若无法支撑业务目标，则拒绝；变更需求不能破坏核心业务目标。

### 二、用户需求：获取方式、表达手段、用例/场景标准写法

#### （1）用户需求如何获取

1. 访谈用户/业务人员（核心手段）；
2. 现场观察现有业务操作；
3. 业务文档、纸质单据分析；
4. 原型演示、问卷调研；
5. 基于业务目标模型定向挖掘模糊、缺失的用户任务。

#### （2）用户需求两种主流表达体系：用例场景模型 / 用户故事

##### 模块A：场景、用例定义与标准表述

###### 1. 场景是什么？怎么写

- **场景定义**：单一、具体的用户与系统交互事件序列，包含上下文、参与者、完整流程，分正常场景、异常/失败场景；
- 核心特点：一条场景只描述**一条完整执行路径**；
- 场景书写模板（取款示例）：

> 场景：正常取款
> 前置上下文：用户已插卡、密码验证通过
> 交互步骤：
> 1. 用户选择取款功能
> 2. 系统弹出金额输入框
> 3. 用户输入取款金额
> 4. 系统校验余额充足，吐出现金
> 5. 用户取走现金
> 6. 系统更新账户余额

- 异常场景示例：余额不足取款，仅修改步骤4及后续提示逻辑。

###### 2. 用例是什么？标准完整表述模板

- **用例定义**：**同一用户目标下所有场景的集合**，收纳正常流程+全部异常分支，是场景的容器；
- 标准用例完整模版（考试必背）：

1. 用例ID、名称、优先级
2. 参与者（主参与者：发起目标；辅助参与者：外部系统/角色）
3. 前置条件（系统进入用例前必须满足的状态）
4. 后置条件（用例执行完成后系统稳定状态）
5. 主成功流程（正常场景，基础步骤）
6. 扩展流程（异常分支，标注对应主流程步骤编号）
7. 特殊需求（非功能：性能、界面、兼容性）
8. 业务规则（计算逻辑、积分规则等）
9. 假设、未解决问题

- 示例：用例「线上购书」整合：正常下单、库存不足、支付失败、中途取消4类场景。

###### 3. 用例与场景核心区别

1. 场景：单一条执行路径（独立故事）；
2. 用例：打包同一个目标下**全部场景**，统一前置/后置条件；
3. 绘制用例图展示用例整体关系，文本用例描述内部场景细节。

##### 模块B：用户故事（敏捷表达用户需求）

1. 标准书写模板（INVEST原则）

> 作为【角色】，我想要【功能动作】，以便获得【业务价值】

2. 三要素：描述语句 + 讨论细节对话 + 验收标准；
3. 组织工具：用户故事地图（主干业务活动→任务→细分故事，划分迭代版本MVP）。

#### （3）用例图四大关系（必考）

1. **包含 «include»**：多个用例共用的公共步骤抽为抽象用例，基用例强制调用；例：下单、退货都包含更新库存；
2. **扩展 «extend»**：不修改原有主用例，新增异常/可选流程，满足触发条件才执行；例：正常销售扩展礼券支付；
3. **泛化 Generalization**：子用例继承父用例全部逻辑，新增专属功能；例：自由查询、列表查询泛化为查询书刊；
4. 参与者泛化：子角色继承父角色所有用例，新增独有用例。

### 三、系统级需求：状态建模、行为建模；系统顺序图 vs 详细顺序图

系统级需求是软件内部交互、数据、逻辑细节，分两大建模维度：**行为建模（顺序图）、状态建模（状态图）**。

#### 1. 行为角度建模：顺序图（核心）

##### （1）两类顺序图核心差异（考试高频考点）

###### ① 系统顺序图（SSD，粗粒度，配合用例）

- 视角：**外部参与者 ↔ 整个系统**，系统作为一个整体方块，不拆分内部Service/Controller；
- 用途：基于用例主流程，梳理用户和系统之间的外部交互消息；
- 特点：不展示系统内部类、分层、组件；仅体现外部输入输出；
- 产出时机：用户需求完成后，初步细化系统交互边界。

###### ② 详细顺序图（内部细粒度）

- 视角：拆分系统内部所有对象/分层（Controller、Service、Repository、DB）；
- 用途：系统详细设计，描述内部对象之间消息传递、循环opt/loop分支；
- 特点：完整分层、展示内部调用逻辑、异常分支、数据流转；
- 产出时机：面向对象分析、系统需求规格编写阶段。

###### 直观对比

| 维度   | 系统顺序图SSD              | 详细顺序图        |
|------|-----------------------|--------------|
| 建模范围 | 外部用户 ↔ 整体系统           | 系统内部所有分层/对象  |
| 粒度   | 粗，无内部细节               | 细，完整内部调用链路   |
| 配合产物 | 用例文本、用例图              | 类图、系统需求规格SRS |
| 分层展示 | 不区分Controller/Service | 完整三层结构展示     |

##### （2）顺序图基础元素

参与者、生命线、同步实线实心箭头、异步虚线箭头、loop循环、opt可选分支、alt多分支。

#### 2. 状态角度建模：状态图

- 建模对象：**单个实体/系统组件**（用户账户、订单、优惠券、红绿灯）；
- 核心表达：对象生命周期内所有状态、触发状态切换的事件、切换后的动作；
- 适用场景：有明确生命周期流转的实体（订单：待支付→已支付→已发货→已完成/取消）；
- 建模逻辑：初始状态 → 多种状态 → 事件触发转移条件 → 终止状态；
- 定位：从**实体生命周期、状态流转**角度描述系统行为，顺序图从**交互消息时序**描述行为，两者互补。

#### 3. 其他系统级建模补充

1. 面向对象：类图（数据结构、实体关系：关联/聚合/组合/泛化）；
2. 结构化分析：数据流图DFD（分层分解数据加工、数据流、存储）；

### 四、全流程文档化手段与规范（分阶段）

#### 1. 业务需求阶段文档

《项目前景与范围文档》：业务总目标、系统边界、涉众、不做的功能；配套目标模型图。

#### 2. 用户需求阶段文档

1. 用例规格说明书：所有用例结构化文本+用例图；
2. 敏捷项目：用户故事地图、用户故事清单、验收标准文档；
3. 场景描述附录：所有正常/异常场景明细。

#### 3. 系统级需求阶段（SRS软件需求规格说明书）

1. 行为模型：系统顺序图、详细顺序图；
2. 状态模型：实体状态图；
3. 数据模型：类图/DFD数据流图；
4. 非功能需求：性能、安全、兼容性；
5. 接口需求：内外系统交互规范。

#### 4. 后期交付文档（第14章）

需求验证报告、需求追溯矩阵（业务目标→用例→系统需求一一对应）、用户手册、测试需求用例。

### 五、完整需求工程执行流程闭环

1. 启动阶段：访谈高层，凝练业务目标，编写《前景范围文档》，构建目标模型；
2. 用户需求获取：基于业务目标访谈用户，收集业务流程；
3. 用户需求建模：梳理场景，整合为用例、绘制用例图；敏捷项目编写用户故事+故事地图；
4. 粗粒度系统行为建模：绘制**系统顺序图**，拆解用户与系统交互；
5. 系统级深度建模：

- 行为：拆分分层，绘制**详细顺序图**；
- 状态：核心实体绘制状态图；
- 数据：类图/DFD；

6. 编写完整SRS系统需求规格说明书；
7. 需求验证：对照业务目标、用例、模型检查完备性、一致性；缺失内容回到需求获取环节补充；
8. 文档固化：生成需求追溯矩阵、交付全套需求文档。
